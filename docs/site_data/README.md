# Manually-Maintained Site Data Files

`site_summary.json` and `graveyard.json` in this directory are **curated by hand**,
not regenerated by the live scheduler (contrast with `ledger_full.json`,
`ledger_recent.json`, `stats.json`, `strategies.json`, which
`nero_core/execution/export_site_data.py` overwrites every run). Update these two only
when a new research batch completes and changes the picture — never auto-scrape them.

## `site_summary.json` — how the numbers were derived (as of 2026-07-18)

**`configs_tested: 67`** — the count of distinct (asset/pair, timeframe, strategy_id +
version) combinations that were actually backtested with real reported trade counts
and expectancy, across every completed research report in `docs/*.md`, with two rules:

1. **Deduplicated across docs.** The same config re-tested in a later, more rigorous
   report (e.g. BNB/12h TREND_PULLBACK v1.0.0 appears in
   `grid_shift_robustness_followup.md`, `statistical_harness_upgrade.md`,
   `trail_exit_ab_report.md`, and `trend_pullback_filter_ab_report.md` as a reference
   baseline) counts ONCE, not once per document.
2. **Excludes data-audit-only investigations** where no strategy was ever backtested —
   `etf_flow_audit.md` (blocked on data) and `white_house_audit.md` (data audit only,
   no registry entry) contribute 0 configs. `nero_analysis.md` and
   `phase0_1_summary.md` are scaffold/analysis docs, not strategy reports.

Breakdown: 2 (MACRO_RISK_ON, BTC+GOLD) + 10 (grid-shift-robustness native configs:
BTC/BNB MEAN_REVERSION-relaxed-pullback, XRP/NEAR MEAN_REVERSION-deep-value,
BNB/12h TREND_PULLBACK, BTC-ETH/12h COINTEGRATION_PAIRS, 4x LEADLAG_FOLLOW pairs)
+ 1 (GOLD BREAKOUT_MOMENTUM gold-calibrated-1week, first CI-harness run) + 8
(FUNDING_EXTREME, 4 assets x 2 timeframes) + 2 (the two EMA-trail v2 variants) + 21
(FVG_REVERSION, 7 assets x 3 timeframes) + 21 (BOS_CONTINUATION, 7 assets x 3
timeframes) + 2 (the two TREND_PULLBACK filter variants) = **67**.

**`strategies_survived: 1`** — configs meeting the strict SURVIVED bar (positive
expectancy on BOTH halves, adequate sample [>=20 trades] on BOTH halves, bootstrap 95%
CI clears zero on BOTH halves), using the LATEST/most rigorous evidence available per
config (a config that passed an earlier, less rigorous check but was later shown to
have a CI crossing zero is NOT counted as survived — the more rigorous result wins).
Only one config meets this bar: **GOLD / 1week / BREAKOUT_MOMENTUM
gold-calibrated-1week v1.2.0** (`docs/statistical_harness_upgrade.md`: train
ExpR 0.395 CI [0.142, 0.648], test ExpR 0.426 CI [0.062, 0.789], both N>=20).
BNB/TREND_PULLBACK and BTC-ETH/COINTEGRATION_PAIRS remain live-scheduler configs (a
separate business decision) but their own CI crosses zero on both halves under the
same re-verification, so neither counts as "survived" here.

Not counted anywhere in `configs_tested` (no dedicated `docs/*.md` report exists for
their own standalone numbers, even though they exist in the codebase/git history):
SHORT_MOMENTUM, DONCHIAN_TREND, the volume-confirmed BREAKOUT_MOMENTUM variant. If a
report for these is ever written, add their configs here.

**`tracking_since: "2026-07-17"`** — the UTC calendar date of the commit that first
deployed `nero_core/execution/live_scheduler.py` (`0a121ef`, 2026-07-18T03:30:52+05:00
= 2026-07-17T22:30:52 UTC).

## `graveyard.json` — sourcing

One entry per distinct killed strategy/config-family, each `source_doc` pointing at
the actual completed report the "why it died" summary is drawn from. Authored once
from the reports existing as of 2026-07-18 — update this file (add entries, don't
rewrite existing ones) after future research batches complete, rather than
auto-scraping `docs/*.md`, since "why it died" is a synthesized judgment call a script
shouldn't be trusted to make unsupervised.
