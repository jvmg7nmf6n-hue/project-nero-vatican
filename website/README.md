# Vatican website

<!-- redeploy trigger -->

Public-facing Next.js site for Project Vatican. Reads the Truth Ledger export
produced by `nero_core/execution/export_site_data.py` (committed to
`docs/site_data/*.json` on the `main` branch) via GitHub raw URLs, with ISR
(`revalidate: 300`). Read-only: this app never writes to the ledger.

## Stack

- Next.js 14 (App Router), React 18
- Tailwind CSS 3
- Jest + React Testing Library

## Local development

```bash
npm install
npm run dev
```

Runs at http://localhost:3000.

## Tests

```bash
npm test
```

Covers: `LedgerTable` empty/populated states and loss styling, the
`deriveCurrentStatus` "current status" derivation logic, and `fetchJson`'s
failure fallback (network error, non-ok response, unparseable body).

> Note: this codebase was authored in an environment without Node.js
> installed, so `npm install` / `npm run build` / `npm test` have not yet
> been run locally. Run all three before or during the first deploy to catch
> anything a compiler or test run would have caught immediately.

## Deploying to Vercel (free tier)

1. Push this repo to GitHub (already the case).
2. In the Vercel dashboard, "Add New Project" → import the
   `project-nero-vatican` repository.
3. Set the project's **Root Directory** to `website/` (the Next.js app does
   not live at the repo root).
4. Framework preset: Next.js (auto-detected). No environment variables are
   required — the site fetches public data from GitHub raw URLs at request
   time, no secrets involved.
5. Deploy. Vercel runs `npm install && npm run build` automatically; this is
   the first real compile check this code will have had.
6. Subsequent pushes to `main` auto-redeploy. Data freshness follows the
   `revalidate: 300` ISR window (5 minutes) independent of redeploys.

## Data contract

This app only reads from `docs/site_data/` at the repo root:

- `ledger_recent.json`, `ledger_full.json`, `strategies.json`, `stats.json` —
  auto-generated by the live scheduler after each run.
- `site_summary.json`, `graveyard.json` — manually curated (see
  `docs/site_data/README.md` for the counting methodology). Update these by
  hand after future research batches; nothing in this app scrapes or
  regenerates them.

All fetch helpers in `lib/data.ts` return `null` on any failure (network
error, non-ok response, bad JSON) so pages can render an honest fallback
instead of crashing.
